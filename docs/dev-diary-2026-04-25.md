# 开发日记 2026-04-25

## 今天做了什么

今天把项目里的语音转录链路从“通义听悟”切到了 `bokegao.pdf` 里描述的 DashScope 方案，并把旧实现彻底清理掉，避免新项目后面再出现双方案并存的问题。

这次改动的核心是三件事：

1. 新增 DashScope 文件上传链路，把本地录音上传到 DashScope 文件管理 OSS，拿到 `oss://...` 内部地址作为 ASR 输入。
2. 新增 DashScope Paraformer 异步转录 provider，负责提交任务、轮询状态、下载 `transcription_url` 结果并整理文本。
3. 删除通义听悟相关 provider、环境变量和引用，让项目默认只有一条转录路径。

## 为什么这样改

PDF 里给出的方案和项目原先的实现不一致。原先代码是“上传到 OSS 公网 URL，再交给听悟离线任务处理”，而 PDF 明确强调了两点：

- 音频转写优先走 DashScope 的 `paraformer-v2`
- 对本地文件优先使用 `oss://` 内部地址，而不是公网 HTTPS 文件地址

如果继续保留听悟实现，短期看似兼容，长期会让配置、排障和后续迭代都变复杂。既然这是新项目，就应该尽早收敛成单一路径。

## 这次落下去的实现

后端现在的转录流程是：

1. 前端上传录音到 API。
2. API 同时做两次上传：
   - 上传到项目自己的 OSS，保留原始录音文件。
   - 上传到 DashScope 文件管理 OSS，拿到 `oss://dashscope-file-mgr/...`。
3. API 使用 DashScope 异步 ASR 创建任务：
   - 模型：`paraformer-v2`
   - 默认语言：`zh,en`
   - 去口语助词：开启
   - 说话人分离：开启
4. API 轮询任务状态，完成后下载 `transcription_url` 对应 JSON。
5. API 将结果整理为可直接使用的 transcript 文本，并写回内存仓库。

另外，为了让数据模型更清晰，我给 `recording` 增加了 `transcriptionSourceUrl` 字段，专门保存提交给 ASR 的内部文件地址，和对外访问的 `audioUrl` 分开。

## 目前仍然存在的风险

虽然代码已经按 PDF 方案接上了，但还有两个现实风险要记下来：

1. DashScope 文件上传依赖 `getPolicy` 接口返回上传凭证，这一步如果后续产品策略或额度策略变化，可能会影响稳定性。
2. 前端采访页目前仍然混有一些旧原型时期的接口调用，转录链路已经独立切好了，但整页业务流后面还需要继续统一到新版 API。
